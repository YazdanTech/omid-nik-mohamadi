DJANGO BACKEND ARCHITECTURE
--------------------------------------------------------------------------------

APP 1: users
    - Models:
        -- CustomUser(AbstractUser):
            --- full_name (CharField)
            --- phone_number (CharField, unique=True)
            --- birth_date (DateField)
            --- email (EmailField, blank=True, null=True)
            --- is_verified (BooleanField, default=False)
        -- SMSVerification:
            --- user (ForeignKey to CustomUser)
            --- code (CharField, max_length=6)
            --- created_at (DateTimeField, auto_now_add=True)
            --- is_used (BooleanField, default=False)
    - Views:
        -- SignUpView (APIView / FormView: validates data, triggers SMS OTP)
        -- VerifyOTPView (Validates code, marks user as verified, logs user in)
        -- LoginView / LogoutView (Session or Token-based auth management)

APP 2: services
    - Models:
        -- Service:
            --- title (CharField)
            --- description (TextField)
            --- price (DecimalField)
            --- duration_minutes (IntegerField)
            --- is_active (BooleanField, default=True)
    - Views:
        -- ServiceListView (Fetches active services and prices for frontend injection)

APP 3: bookings
    - Models:
        -- BookingSlot:
            --- date (DateField)
            --- start_time (TimeField)
            --- is_booked (BooleanField, default=False)
        -- BypassCode:
            --- code (CharField, unique=True)
            --- description (CharField)
            --- is_active (BooleanField, default=True)
        -- Booking:
            --- user (ForeignKey to CustomUser)
            --- service (ForeignKey to Service)
            --- slot (OneToOneField to BookingSlot)
            --- deposit_paid (BooleanField, default=False)
            --- bypass_code_used (ForeignKey to BypassCode, null=True, blank=True)
            --- status (CharField: Pending, Confirmed, Cancelled)
    - Views:
        -- AvailableSlotsView (Returns unbooked date/time matrix filtered by date)
        -- CreateBookingView (Process code bypass or triggers $500 payment logic)

APP 4: shop
    - Models:
        -- Product:
            --- title (CharField)
            --- description (TextField)
            --- price (DecimalField)
            --- stock (IntegerField)
            --- image (ImageField)
        -- Order:
            --- user (ForeignKey to CustomUser)
            --- total_amount (DecimalField)
            --- created_at (DateTimeField)
            --- is_paid (BooleanField, default=False)
        -- OrderItem:
            --- order (ForeignKey to Order)
            --- product (ForeignKey to Product)
            --- quantity (IntegerField)
    - Views:
        -- ProductCatalogView (Returns all products)
        -- CheckoutView (Handles order generation and inventory decrement)

================================================================================
3. GIT COMMIT ARCHITECTURE
================================================================================

Keep commits standardized but simple using concise functional prefixes.

PREFIXES:
    - `feat:`     -> Introducing a brand new programming feature or model.
    - `fix:`      -> Rectifying a codebase bug or design broken asset link.
    - `style:`    -> Code changes strictly involving CSS, layout spacing, colors, or themes.
    - `docs:`     -> Writing readmes, architecture guidelines, or configuration comments.
    - `refactor:` -> Code modification that neither fixes a bug nor adds a feature.

RULES:
    - Write in the present imperative tense ("add", not "added").
    - Maximum 50 characters for the subject title line.

EXAMPLES:
    - `feat: implement custom user model with phone authentication`
    - `style: establish luxury dark theme CSS variables and global typography`
    - `fix: correct booking date matrix timezone offset query bug`
    - `feat: build deposit payment requirement verification logic`
    - `docs: update deployment asset collection requirements checklist`
